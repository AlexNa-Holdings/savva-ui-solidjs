# Ініціалізація застосунку та підключення до бекенду

## Що таке бекенд SAVVA (вузол SAVVA)?
Бекенд SAVVA — це серверний компонент, який **індексує/кешує дані, отримані з активності блокчейну**, і надає швидкі, зручні для інтерфейсу користувача API та WebSocket-методи. Один бекенд може обслуговувати **кілька доменів SAVVA** — уявіть «домен» як окрему соціальну мережу SAVVA (брендинг, вкладки, ресурси, налаштування за замовчуванням), всі вони працюють на одному вузлі.

## Що потрібно застосунку під час запуску
Під час старту вебзастосунку потрібні два параметри:

1. **Backend URL** – базовий URL бекенду SAVVA.
2. **Domain name** – який домен SAVVA (соціальна мережа) відображати за замовчуванням.

Значення за замовчуванням беруться з невеликого JSON-файлу в корені вебу (YAML також підтримується як запасний варіант):

### `/default_connect.json`
```json
{
  "domain": "savva.app",
  "backendLink": "https://ui.savva.app/api/",
  "gear": true,
  "default_ipfs_link": "ipfs://bafy.../something.json"
}
```

* `backendLink` — базовий HTTP-ендпоінт бекенду SAVVA (застосунок нормалізує його).
* `domain` — початковий домен для відображення; його можна переключити пізніше в UI.
* `gear` — вмикає інструменти розробника в інтерфейсі (необов’язково).
* `default_ipfs_link` — необов’язковий зручний дефолт, який використовується в деяких потоках.

> **Примітка щодо формату**
> Застосунок спочатку намагається отримати `/default_connect.json`. Якщо цей запит не вдається, він відкочується до `/default_connect.yaml` для зворотної сумісності. Нові розгортання мають використовувати JSON.

> **Примітка для продакшену**
> У продакшені цей файл зазвичай обслуговує ваш HTTP-сервер (наприклад, Nginx) і фактично **визначає, який домен** буде показано вебзастосунком за замовчуванням. Один із поширених підходів — віддавати конкретний файл з диска:
>
> ```nginx
> # example: serve a static default_connect.json
> location = /default_connect.json {
>   default_type application/json;
>   alias /etc/savva/default_connect.json;
> }
> ```
>
> Налаштуйте під вашу інфраструктуру; важливо, щоб застосунок міг робити `GET /default_connect.json`.

## Послідовність завантаження

1. **Завантажити конфіг сайту (`/default_connect.json` або `.yaml`)**
   Застосунок спочатку намагається отримати `/default_connect.json`; якщо недоступний — відкочується до `/default_connect.yaml`. Він перевіряє `backendLink`, зберігає `domain` і негайно **налаштовує ендпоінти** (HTTP-базу + WS-URL) на основі цих значень. &#x20;

2. **Налаштувати ендпоінти**

   * `httpBase` — це нормалізована версія `backendLink` (гарантовано з кінечною косою рискою).
   * `ws` URL походить від тієї самої бази, вказуючи на `.../ws` (протокол змінюється на `ws:` або `wss:`) і включає `?domain=...` у запиті.
     Це зберігає **одне джерело правди** для HTTP і WS.&#x20;

3. **Отримати `/info`**
   Маєчи налаштовані ендпоінти, застосунок виконує `GET <httpBase>info` і зберігає JSON. Від цього моменту **/info визначає поведінку під час виконання** (домени, ланцюг, IPFS, ресурси).&#x20;

4. **Вивести стан виконання з `/info`**
   Використовуються наступні поля (див. приклад нижче):

   * **`domains`** → список доступних доменів. UI віддає перевагу явному `domain` з YAML/перезапису; якщо його немає в `/info`, застосунок все одно використовує налаштований домен.&#x20;
   * **`blockchain_id`** → цільовий ID ланцюга EVM. Помічник для гаманця може переключити/додати цю мережу.&#x20;
   * **`ipfs_gateways`** → віддалені IPFS‑шлюзи, які слід пробувати в порядку черги (якщо не ввімкнено локальне перекриття IPFS).&#x20;
   * **`assets_url`** та **`temp_assets_url`** → **база ресурсів** (прод/тест). Застосунок обчислює **префікс активних ресурсів домену** як
     `(<assets base> + <domain> + "/")` з **запасним варіантом** `/domain_default/`, якщо віддалений `config.yaml` відсутній. &#x20;

5. **Завантажити ресурси та конфіг домену**
   Застосунок пробує `(<active prefix>/config.yaml)` з коротким таймаутом; у разі невдачі переходить до пакету за замовчуванням `/domain_default/config.yaml`. Отриманий розпарсений конфіг (логотипи, вкладки, локалі тощо) зберігається, і UI рендериться відповідно.&#x20;

6. **WebSocket під час виконання**
   WS-клієнт використовує обчислений `ws` URL з ендпоінтів; коли бекенд/домен змінюється, ендпоінти перераховуються, і WS-шар підхоплює це.&#x20;

## Приклад `/info` (ілюстративно)

```json
{
  "domains": [
    "savva.app",
    {"name": "art.savva"},
    "dev.savva"
  ],
  "blockchain_id": 369,
  "ipfs_gateways": [
    "https://cloudflare-ipfs.com/ipfs/",
    "https://ipfs.io/ipfs/"
  ],
  "assets_url": "https://cdn.savva.network/assets/",
  "temp_assets_url": "https://cdn.savva.network/assets-test/"
}
```

### Поле за полем (що з цим робить застосунок)

* **domains** — список вибіркових доменів. Діалог **Switch backend / domain** заповнюється з `/info`, але налаштований домен все ще має пріоритет, якщо `/info` відстає. &#x20;
* **blockchain\_id** — числовий ID ланцюга EVM; використовується для формування метаданих `switch/add chain` і для забезпечення того, щоб гаманець був у **потрібній мережі**. &#x20;
* **ipfs\_gateways** — впорядкований список віддалених шлюзів; комбінується з необов’язковим **локальним IPFS** (коли увімкнено в налаштуваннях) для формування **активного** порядку шлюзів.&#x20;
* **assets\_url / temp\_assets\_url** — застосунок підтримує середовище ресурсів (`prod`/`test`) і обирає відповідну базу. Потім він обчислює `/<base>/<domain>/` і завантажує `config.yaml`. Якщо віддалений пакет відсутній або повільний, використовується **дефолт** `/domain_default/`.&#x20;

## Де це знаходиться в коді (для швидкого ознайомлення)

* Завантаження при старті та конфіг сайту (`/default_connect.json` з `.yaml` fallback), потім `/info`: **`src/context/AppContext.jsx`** та **`src/hooks/useConnect.js`**. Спільний завантажувач знаходиться в **`src/utils/loadSiteConfig.js`**. &#x20;
* Джерело істини для ендпоінтів (HTTP‑база + WS URL): **`src/net/endpoints.js`**.&#x20;
* Розв'язування списку доменів, ID ланцюга, IPFS-шлюзів, середовище ресурсів та завантаження ресурсів домену: **`src/context/AppContext.jsx`**.  &#x20;
* Діалог перемикання, який отримує `/info` і нормалізує `domains`: **`src/x/SwitchConnectModal.jsx`**.&#x20;

---

**Далі:** у наступній главі ми розберемо **конфігурацію домену** (`config.yaml`) і як вона керує логотипами, вкладками, локалями та іншими поведінками UI на рівні домену.
